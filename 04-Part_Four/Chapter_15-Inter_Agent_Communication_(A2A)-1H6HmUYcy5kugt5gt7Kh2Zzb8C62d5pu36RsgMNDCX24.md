# Глава 15: Межагентная коммуникация (A2A)

Отдельные AI-агенты часто сталкиваются с ограничениями при решении сложных многоаспектных задач, даже обладая продвинутыми возможностями. Для преодоления этого межагентная коммуникация (Inter-Agent Communication, A2A) позволяет различным AI-агентам, потенциально построенным на разных фреймворках, эффективно сотрудничать. Это сотрудничество включает seamless координацию, делегирование задач и обмен информацией.

Протокол A2A от Google — это открытый стандарт, предназначенный для обеспечения универсальной коммуникации. В этой главе мы рассмотрим A2A, его практические применения и реализацию в Google ADK.

## Обзор паттерна «Межагентная коммуникация»

Протокол Agent2Agent (A2A) — открытый стандарт, предназначенный для обеспечения коммуникации и сотрудничества между различными фреймворками AI-агентов. Он гарантирует совместимость, позволяя AI-агентам, разработанным с использованием технологий вроде LangGraph, CrewAI или Google ADK, работать совместно независимо от происхождения или различий фреймворков.

A2A поддерживается рядом технологических компаний и провайдеров услуг, включая Atlassian, Box, LangChain, MongoDB, Salesforce, SAP и ServiceNow. Microsoft планирует интегрировать A2A в Azure AI Foundry и Copilot Studio, демонстрируя приверженность открытым протоколам. Кроме того, Auth0 и SAP интегрируют поддержку A2A в свои платформы и агенты.

Как открытый протокол, A2A приветствует вклад сообщества для содействия его развитию и широкому распространению.

## Базовые концепции A2A

Протокол A2A предоставляет структурированный подход к взаимодействию агентов, основанный на нескольких ключевых концепциях. Глубокое понимание этих концепций критически важно для всех, кто разрабатывает или интегрирует A2A-совместимые системы. Фундаментальные столпы A2A включают основных субъектов, Agent Card, обнаружение агентов, коммуникацию и задачи, механизмы взаимодействия и безопасность.

**Основные субъекты (Core Actors):** A2A включает три основные сущности:

* **User (пользователь):** Инициирует запросы на помощь агента.
* **A2A Client (клиентский агент):** Приложение или AI-агент, действующий от имени пользователя для запроса действий или информации.
* **A2A Server (удалённый агент):** AI-агент или система, предоставляющий HTTP-эндпоинт для обработки запросов клиентов и возврата результатов. Удалённый агент работает как «непрозрачная» система, то есть клиенту не нужно понимать детали его внутренней работы.

**Agent Card:** Цифровая идентичность агента определяется его Agent Card — обычно JSON-файлом. Этот файл содержит ключевую информацию для взаимодействия клиентов и автоматического обнаружения, включая идентичность агента, URL эндпоинта и версию. Он также детализирует поддерживаемые возможности вроде streaming или push-уведомлений, конкретные навыки, режимы ввода/вывода по умолчанию и требования аутентификации. Ниже приведён пример Agent Card для WeatherBot.

**Обнаружение агентов (Agent discovery):** позволяет клиентам находить Agent Cards, которые описывают возможности доступных A2A Servers. Существует несколько стратегий этого процесса:

* **Well-Known URI:** Агенты размещают свой Agent Card по стандартизированному пути (например, `/.well-known/agent.json`). Этот подход обеспечивает широкую, часто автоматизированную доступность для публичных или доменно-специфичных сценариев.
* **Курируемые реестры:** Предоставляют централизованный каталог, куда публикуются Agent Cards и могут запрашиваться на основе конкретных критериев. Хорошо подходит для корпоративных сред, требующих централизованного управления и контроля доступа.
* **Прямая конфигурация:** Информация Agent Card встраивается или передаётся приватно. Этот метод подходит для closely связанных или приватных систем, где динамическое обнаружение не критично.

Независимо от выбранного метода важно защищать эндпоинты Agent Card. Это может достигаться через контроль доступа, взаимный TLS (mTLS) или сетевые ограничения, особенно если карта содержит конфиденциальную (хотя и не секретную) информацию.

**Коммуникация и задачи:** В фреймворке A2A коммуникация структурирована вокруг асинхронных задач (tasks), которые представляют фундаментальные единицы работы для длительных процессов. Каждой задаче присваивается уникальный идентификатор, и она проходит через серию состояний — таких как submitted, working или completed. Такой дизайн supports параллельную обработку в сложных операциях. Коммуникация между агентами происходит через Message (сообщение), которое содержит атрибуты — метаданные в формате ключ-значение, описывающие приоритет или время создания сообщения, — и одну или несколько частей (parts), несущих фактическое доставляемое содержимое: обычный текст, файлы или структурированные JSON-данные. Ощутимые выходные данные, генерируемые агентом во время выполнения задачи, называются артефактами (artifacts). Как и сообщения, артефакты состоят из одной или нескольких частей и могут streamed инкрементально по мере появления результатов. Вся коммуникация в фреймворке A2A ведётся по протоколу HTTP(S) с использованием JSON-RPC 2.0 для формирования нагрузок. Для поддержания непрерывности across множественных взаимодействий используется сервер-генерируемый идентификатор contextId, позволяющий группировать связанные задачи и сохранять контекст throughout сессии.

**Механизмы взаимодействия:** A2A предоставляет несколько методов взаимодействия для различных потребностей AI-приложений, каждый с отличительным механизмом:

* **Синхронный запрос/ответ:** Для быстрых, немедленных операций. Клиент отправляет запрос и активно ожидает, пока сервер обработает его и вернёт полный ответ в одном синхронном обмене.
* **Асинхронный опрос:** Подходит для задач, требующих более длительной обработки. Клиент отправляет запрос, сервер немедленно подтверждает его со статусом «working» и ID задачи. Клиент затем может выполнять другие действия и периодически опрашивать сервер, проверяя статус задачи до её завершения или сбоя.
* **Потоковые обновления (Server-Sent Events — SSE):** Идеально для получения результатов в реальном времени. Устанавливает постоянное однонаправленное соединение от сервера к клиенту, позволяя удалённому агенту непрерывно pushing обновления — изменения статуса или частичные результаты — без необходимости множественных запросов клиента.
* **Push-уведомления (Webhooks):** Предназначены для очень длительных или ресурсоёмких задач, где поддержание постоянного соединения или частый опрос неэффективны. Клиент может зарегистрировать URL webhook, и сервер отправит асинхронное уведомление при значительном изменении статуса задачи (например, при завершении).

Agent Card указывает, поддерживает ли агент streaming или push-уведомления. Кроме того, A2A модально-agnostic, что означает возможность содействия этим паттернов взаимодействия не только для текста, но и для других типов данных — аудио и видео, обеспечивая насыщенные мультимодальные AI-приложения.
```json
# Пример синхронного запроса
{
    "jsonrpc": "2.0",
    "id": "1",
    "method": "sendTask",
    "params": {
        "id": "task-001",
        "sessionId": "session-001",
        "message": {
            "role": "user",
            "parts": [
                {
                    "type": "text",
                    "text": "What is the exchange rate from USD to EUR?"
                }
            ]
        },
        "acceptedOutputModes": [
            "text/plain"
        ],
        "historyLength": 5
    }
}
```
Синхронный запрос использует метод sendTask, где клиент запрашивает и ожидает единый и полный ответ на свой запрос. В contrast, стриминговый запрос использует метод sendTaskSubscribe для установления постоянного соединения, позволяя агенту отправлять multiple инкрементальные обновления или частичные результаты с течением времени.
```json
# Пример стримингового запроса
{
    "jsonrpc": "2.0",
    "id": "2",
    "method": "sendTaskSubscribe",
    "params": {
        "id": "task-002",
        "sessionId": "session-001",
        "message": {
            "role": "user",
            "parts": [
                {
                    "type": "text",
                    "text": "What's the exchange rate for JPY to GBP today?"
                }
            ]
        },
        "acceptedOutputModes": [
            "text/plain"
        ],
        "historyLength": 5
    }
}
```
**Безопасность:** Межагентная коммуникация (A2A) является жизненно важным компонентом системной архитектуры, обеспечивающим безопасный и бесшовный обмен данными между агентами. Она гарантирует надёжность и целостность через несколько встроенных механизмов, each designed addressing различные аспекты защиты.

**Взаимный TLS (Mutual Transport Layer Security):** Устанавливаются зашифрованные и взаимно аутентифицированные соединения между агентами для предотвращения несанкционированного доступа и перехвата данных. Это ensures что обе стороны коммуникации верифицированы, обеспечивая безопасный канал обмена информацией.

**Детальные журналы аудита:** Все межагентные коммуникации детально записываются с указанием потока информации, involved агентов и выполненных действий. Этот аудиторский след является критически важным для подотчётности, устранения неполадок и анализа безопасности, позволяя администраторам отследить любое взаимодействие retroactively.

**Декларация Agent Card:** Требования аутентификации явно указываются в Agent Card — конфигурационном артефакте, определяющем идентичность, возможности и политики безопасности агента. Такой подход централизует и упрощает управление аутентификацией, позволяя клиентам сразу понимать требования безопасности перед началом взаимодействия.

**Обработка учётных данных:** Агенты обычно аутентифицируются с использованием безопасных учётных данных вроде OAuth 2.0 токенов или API ключей, передаваемых через HTTP-заголовки. Этот метод предотвращает раскрытие учётных данных в URL-адресах или телах сообщений, значительно повышая общую безопасность системы. Рекомендуется использовать короткоживущие токены с регулярным обновлением для минимизации рисков компрометации.

## A2A против MCP

A2A — протокол, дополняющий Model Context Protocol (MCP) от Anthropic (см. Рис. 1). В то время как MCP фокусируется на структурировании контекста для агентов и их взаимодействии с внешними данными и инструментами, A2A содействует координации и коммуникацию между агентами, обеспечивая делегирование задач и сотрудничество.

Цель A2A — повышение эффективности, снижение затрат на интеграцию и fostering инноваций и совместимости в разработке сложных многоагентных AI-систем. Поэтому thorough понимание ключевых компонентов и методов работы A2A необходимо для эффективного проектирования, реализации и применения в построении совместных и interoperable AI-агентных систем.

## Практические применения и сценарии использования

Межагентная коммуникация незаменима для построения утончённых AI-решений в различных доменах, обеспечивая модульность, масштабируемость и улучшенный интеллект.

* **Многофреймворковое сотрудничество:** Основной сценарий использования A2A — обеспечение коммуникации и сотрудничества независимых AI-агентов regardless их underlying фреймворков (например, ADK, LangChain, CrewAI). Это фундаментально для построения сложных многоагентных систем, где различные агенты специализируются на различных аспектах задачи.
* **Оркестрация автоматизированных рабочих процессов:** В корпоративных средах A2A содействует сложным рабочие процессы, позволяя агентам делегировать и координировать задачи. Например, агент может управлять первоначальным сбором данных, затем делегировать другому агенту анализ и, наконец, третьему — генерацию отчёта, все communicating через протокол A2A.
* **Динамическое извлечение информации:** Агенты могут communicate для извлечения и обмена информацией в реальном времени. Первичный агент может запрашивать оперативные рыночные данные у специализированного «агента fetching данных», который затем использует внешние API для сбора информации и отправляет её обратно.

## Практический пример кода

```python
import datetime

from google.adk.agents import LlmAgent  # type: ignore[import-untyped]
from google.adk.tools.google_api_tool import CalendarToolset  # type: ignore[import-untyped]


async def create_agent(client_id: str, client_secret: str) -> LlmAgent:
    """Создаёт ADK-агента."""
    toolset = CalendarToolset(client_id=client_id, client_secret=client_secret)
    return LlmAgent(
        model="gemini-2.0-flash-001",
        name="calendar_agent",
        description="An agent that can help manage a user's calendar",
        instruction=(
            f""" You are an agent that can help manage a user's calendar. Users will request information about the state of their calendar """
            f""" or to make changes to their calendar. Use the provided tools for interacting with the calendar API. """
            f""" If not specified, assume the calendar the user wants is the 'primary' calendar. """
            f""" When using the Calendar API tools, use well-formed RFC3339 timestamps. Today is {datetime.datetime.now()}. """
        ),
        tools=await toolset.get_tools(),
    )
```
Этот код на Python определяет асинхронную функцию `create_agent`, которая конструирует ADK LlmAgent. Она начинает с инициализации `CalendarToolset` с использованием предоставленных учётных данных клиента (client ID и client secret) для доступа к Google Calendar API. Затем создаётся экземпляр `LlmAgent`, настроенный с указанной моделью Gemini, описательным именем и детализированными инструкциями по управлению календарём пользователя. Агент оснащается инструментами календаря из `CalendarToolset`, позволяя ему взаимодействовать с Calendar API и отвечать на запросы пользователей о текущих состояниях календаря или запрашиваемых модификациях. Инструкции агента динамически включающую текущую дату для обеспечения релевантного временного контекста. Чтобы проиллюстрировать, как конструируется агент, рассмотрим ключевую секцию из `calendar_agent`, найденного в примерах A2A на GitHub. Ниже приведён код, показывающий определение агента с конкретными инструкциями и инструментами. Обратите внимание, что показан только код, необходимый для объяснения данной функциональности; полный файл доступен по ссылке.
```python
def main(host: str = "0.0.0.0", port: int = 8000):
    # Проверка наличия API ключа.
    # Не требуется при использовании Vertex AI API.
    if os.getenv("GOOGLE_GENAI_USE_VERTEXAI") != "TRUE" and not os.getenv("GOOGLE_API_KEY"):
        raise ValueError(
            "GOOGLE_API_KEY environment variable not set and "
            "GOOGLE_GENAI_USE_VERTEXAI is not TRUE."
        )

    skill = AgentSkill(
        id="check_availability",
        name="Check Availability",
        description="Checks a user's availability for a time using their Google Calendar",
        tags=["calendar"],
        examples=["Am I free from 10am to 11am tomorrow?"],
    )

    agent_card = AgentCard(
        name="Calendar Agent",
        description="An agent that can manage a user's calendar",
        url=f"http://{host}:{port}/",
        version="1.0.0",
        defaultInputModes=["text"],
        defaultOutputModes=["text"],
        capabilities=AgentCapabilities(streaming=True),
        skills=[skill],
    )

    adk_agent = asyncio.run(
        create_agent(
            client_id=os.getenv("GOOGLE_CLIENT_ID"),
            client_secret=os.getenv("GOOGLE_CLIENT_SECRET"),
        )
    )

    runner = Runner(
        app_name=agent_card.name,
        agent=adk_agent,
        artifact_service=InMemoryArtifactService(),
        session_service=InMemorySessionService(),
        memory_service=InMemoryMemoryService(),
    )
    agent_executor = ADKAgentExecutor(runner, agent_card)

    async def handle_auth(request: Request) -> PlainTextResponse:
        await agent_executor.on_auth_callback(
            str(request.query_params.get("state")),
            str(request.url),
        )
        return PlainTextResponse("Authentication successful.")

    request_handler = DefaultRequestHandler(
        agent_executor=agent_executor,
        task_store=InMemoryTaskStore(),
    )

    a2a_app = A2AStarletteApplication(
        agent_card=agent_card,
        http_handler=request_handler,
    )
    routes = a2a_app.routes()
    routes.append(
        Route(
            path="/authenticate",
            methods=["GET"],
            endpoint=handle_auth,
        )
    )
    app = Starlette(routes=routes)

    uvicorn.run(app, host=host, port=port)


if __name__ == "__main__":
    main()
```
Этот код на Python демонстрирует настройку A2A-совместимого «Calendar Agent» для проверки доступности пользователя с использованием Google Calendar. Он включает проверку API ключей или конфигураций Vertex AI для аутентификации. Возможности агента, включая навык «check_availability», определяются в AgentCard, которая также указывает сетевой адрес агента. Затем создаётся ADK-агент, настроенный с сервисами in-memory для управления артефактами, сессиями и памятью. Код инициализирует веб-приложение Starlette, включает callback аутентификации и обработчик протокола A2A, и запускает его с помощью Uvicorn, exposing агент через HTTP.

Эти примеры illustrate процесс построения A2A-совместимого агента — от определения его возможностей до запуска в качестве веб-сервиса. Используя Agent Cards и ADK, разработчики могут создавать interoperable AI-агентов, capable интеграции с инструментами вроде Google Calendar. Этот practical подход демонстрирует применение A2A в establishing многоагентной экосистемы.

## Краткий обзор

**Что:** Отдельные AI-агенты, особенно построенные на разных фреймворках, часто испытывают трудности со сложными многоаспектными задачами в одиночку. Основная проблема — отсутствие общего языка или протокола, позволяя им эффективно взаимодействовать и сотрудничать. Эта изоляция препятствует созданию утончённых систем, где multiple специализированные агенты могут комбинировать свои уникальные навыки для решения более крупных задач. Без стандартизированного подхода интеграция различных агентов costly, time-consuming и тормозит разработку более мощных, cohesive AI-решений.

**Почему:** Протокол межагентной коммуникации (A2A) provides открытое стандартизированное решение этой проблемы. Это HTTP-протокол, обеспечивающий совместимость, позволяя различным AI-агентам координировать, делегировать задачи и обмениваться информацией бесшовно независимо от их базовых технологии. Ключевой компонент — Agent Card, цифровой идентификационный файл, описывающий возможности, навыки и эндпоинты коммуникации агента, содействуя обнаружению и взаимодействие. A2A определяет различные механизмы взаимодействия, включая синхронную и асинхронную коммуникацию, для поддержки различных сценариев использования. Создавая универсальный стандарт для сотрудничества агентов, A2A формирует модульную и масштабируемую экосистему для построения сложных многоагентных Agentic систем.

**Когда использовать:** Используйте этот паттерн, когда необходимо оркестрировать сотрудничество двух или более AI-агентов, especially если они построены на разных фреймворках (например, Google ADK, LangGraph, CrewAI). Он идеален для построения сложных модульных приложений, где специализированные агенты управляют конкретными частями рабочего процесса — таких как делегирование анализа данных одному агенту и генерации отчётов другому. Этот паттерн также необходим, когда агенту нужно динамически обнаруживать и использовать возможности других агентов для выполнения задачи.

## Ключевые выводы

* Протокол A2A от Google — открытый HTTP-стандарт, содействующий коммуникации и сотрудничество между AI-агентами, построенными на разных фреймворках.
* AgentCard выступает в качестве цифрового идентификатора агента, позволяя автоматическое обнаружение и понимание его возможностей другими агентами.
* A2A offers как синхронные запросы-ответы (с использованием `tasks/send`), так и стриминговые обновления (с использованием `tasks/sendSubscribe`) для various потребностей коммуникации.
* Протокол поддерживает многоходовые беседы, включая состояние `input-required`, позволяя агентам запрашивать дополнительную информацию и поддерживать контекст во время взаимодействий.
* A2A поощряет модульную архитектуру, где специализированные агенты могут работать автономно на различных портах, обеспечивая масштабируемость и распределённость системы.
* Инструменты вроде Trickle AI aid в визуализации и отслеживании A2A-коммуникаций, helping разработчикам monitor, debug и оптимизировать многоагентные системы.
* В то время как A2A — высокоуровневый протокол для управления задачами и рабочими процессами между агентами, Model Context Protocol (MCP) provides стандартизированный интерфейс для LLM к взаимодействию с внешними ресурсами.

## Заключение

Протокол межагентной коммуникации (A2A) устанавливает vital открытый стандарт для преодоления inherent изоляции отдельных AI-агентов. Предоставляя общий HTTP-фреймворк, он обеспечивает бесшовное сотрудничество и совместимость между агентами, построенными на различных платформах — Google ADK, LangGraph или CrewAI. Ключевой компонент — Agent Card, выступая в качестве цифровой идентичности, чётко определяя возможности агента и обеспечивая динамическое обнаружение другими агентами. Гибкость протокола поддерживает различные паттерны взаимодействия — синхронные запросы, асинхронный опрос и стриминг в реальном времени — охватывая широкий спектру потребностей приложений.

Это обеспечивает создание модульных и масштабируемых архитектур, где специализированные агенты могут комбинироваться для оркестрации сложных автоматизированных рабочих процессов. Безопасность — фундаментальный аспект со встроенными механизмами, такими как mTLS и явных требований аутентификации для защиты коммуникаций. Дополняя другие стандарты вроде MCP, уникальный фокус A2A — высокоуровневая координация и делегирование задач между агентами. Сильная поддержка крупных технологических компаний и доступность практических реализаций подчёркивая растущее значение протокола. Этот протокол прокладывает путь разработчикам для построения более утончённых, распределённых и интеллектуальных многоагентных систем. В конечном счёте A2A — фундаментальный столп формирования инновационной и interoperable экосистемы совместного AI.
