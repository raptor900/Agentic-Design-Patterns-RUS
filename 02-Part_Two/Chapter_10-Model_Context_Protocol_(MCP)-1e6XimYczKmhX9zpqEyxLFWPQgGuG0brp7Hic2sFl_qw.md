# Глава 10: Протокол контекста модели (MCP)

Чтобы LLM эффективно работали как агенты, их возможности должны выходить за рамки мультимодальной генерации. Необходимо взаимодействие с внешней средой: доступ к текущим данным, использование внешнего ПО и выполнение операционных задач. Протокол контекста модели (Model Context Protocol, MCP) решает эту задачу, предоставляя стандартизированный интерфейс для связи LLM с внешними ресурсами.

## Обзор паттерна MCP

Представьте себе универсальный адаптер, позволяющий любой LLM подключиться к любой внешней системе, БД или инструменту без кастомной интеграции. По сути, это и есть MCP — открытый стандарт для унификации коммуникации LLM (Gemini, GPT, Mixtral, Claude) с внешними приложениями, источниками данных и инструментами.

MCP работает по клиент-серверной архитектуре. Он определяет, как элементы — данные (resources), интерактивные шаблоны (prompts) и функции (tools) — экспонируются MCP-сервером и потребляются MCP-клиентом (LLM-хост или AI-агент).

Однако MCP — это контракт для «агентного интерфейса», и его эффективность зависит от дизайна underlying API. Существует риск, что разработчики просто оборачивают legacy API без модификации, что может быть субоптимально для агента. Например, если API позволяет извлекать только полные данные по одному элементу, агент будет медленным при больших объёмах. Для эффективности underlying API должен быть улучшен детерминированными функциями (фильтрация, сортировка).

## MCP vs. Вызов функций инструментов

MCP и вызов функций — различные механизмы взаимодействия LLM с внешними возможностями. Оба расширяют LLM за пределы генерации текста, но различаются подходом.

| Возможность | Вызов функций | MCP |
|---|---|---|
| **Стандартизация** | Проприетарный, зависит от вендора | Открытый стандарт |
| **Область** | Прямой механизм вызова конкретной функции | Широкий фреймворк для обнаружения и коммуникации |
| **Архитектура** | Взаимодействие «один-к-одному» | Клиент-серверная |
| **Обнаружение** | LLM явно told об инструментах | Динамическое обнаружение |
| **Переиспользуемость** | Привязка к конкретному приложению | Автономные серверы |

MCP — как универсальная розетка: не предоставляет инструменты, но позволяет любому совместимому инструменту подключаться и работать.

## Дополнительные соображения по MCP

* **Tool vs. Resource vs. Prompt**: Resource — статические данные, Tool — исполняемая функция, Prompt — шаблон для взаимодействия LLM с ресурсом или инструментом.
* **Обнаружаемость**: MCP-клиент может динамически запрашивать сервер о доступных возможностях.
* **Безопасность**: Аутентификация и авторизация для контроля доступа.
* **Реализация**: Поставщики (Anthropic, FastMCP) упрощают процесс через SDK.
* **Обработка ошибок**: Протокол определяет, как ошибки передаются LLM.
* **Локальный vs. удалённый сервер**: Локальный — для скорости и безопасности, удалённый — для масштабируемости.
* **По запросу vs. пакетная обработка**: Оба режима поддерживаются.
* **Транспортный механизм**: JSON-RPC через STDIO (локально), Streamable HTTP и SSE (удалённо).

Взаимодействие компонентов MCP:

1. **LLM**: Основной интеллект — обрабатывает запросы, формулирует планы, решает когда обратиться к внешнему ресурсу.
2. **MCP Client**: Приложение-обёртка LLM, транслирующее намерения в MCP-запросы.
3. **MCP Server**: Шлюз к внешнему миру — экспонирует инструменты, ресурсы и промпты.
4. **Сторонний сервис**: Фактический внешний инструмент, приложение или источник данных.

Поток взаимодействия:
1. **Обнаружение**: Клиент запрашивает сервер о доступных возможностях.
2. **Формулирование запроса**: LLM определяет необходимый инструмент и параметры.
3. **Коммуникация**: Клиент отправляет стандартизированный вызов серверу.
4. **Выполнение**: Сервер аутентифицирует, валидирует и выполняет действие.
5. **Ответ**: Результат возвращается LLM для обновления контекста.

## Практические применения и сценарии использования

* **Интеграция с БД**: Доступ к структурированным данным (BigQuery) через естественный язык.
* **Оркестрация генеративных медиа**: Интеграция с Imagen, Veo, Chirp 3 HD, Lyria.
* **Взаимодействие с внешними API**: Погода, биржевые котировки, email, CRM.
* **Извлечение информации на основе рассуждений**: Точный поиск по документам.
* **Разработка пользовательских инструментов**: FastMCP для экспонирования внутренних функций.
* **Стандартизированная коммуникация LLM-приложение**: Снижение накладных расходов на интеграцию.
* **Оркестрация сложных рабочих процессов**: Многошаговые процессы с несколькими MCP-сервисами.
* **Управление IoT-устройствами**: Естественно-языковое управление умным домом.
* **Автоматизация финансовых сервисов**: Анализ данных, торговля, отчётность.

## Практический пример кода с ADK

Подключение к локальному MCP-серверу файловой системы:

```python
import os
from google.adk.agents import LlmAgent
from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, StdioServerParameters

TARGET_FOLDER_PATH = os.path.join(
    os.path.dirname(os.path.abspath(__file__)),
    "mcp_managed_files",
)

os.makedirs(TARGET_FOLDER_PATH, exist_ok=True)

root_agent = LlmAgent(
    model="gemini-2.0-flash",
    name="filesystem_assistant_agent",
    instruction=(
        "Help the user manage their files. You can list files, read files, and write files. "
        f"You are operating in the following directory: {TARGET_FOLDER_PATH}"
    ),
    tools=[
        MCPToolset(
            connection_params=StdioServerParameters(
                command="npx",
                args=[
                    "-y",
                    "@modelcontextprotocol/server-filesystem",
                    TARGET_FOLDER_PATH,
                ],
            ),
        )
    ],
)
```

npx (Node Package Execute) — утилита для прямого выполнения Node.js-пакетов из реестра npm без глобальной установки. Используется для запуска MCP-серверов.

## Создание MCP-сервера с FastMCP

FastMCP — высокоуровневый Python-фреймворк для упрощённой разработки MCP-серверов. Автоматически генерирует схемы из сигнатур функций, аннотаций типов и docstring.

```python
from fastmcp import FastMCP

mcp_server = FastMCP()

@mcp_server.tool
def greet(name: str) -> str:
    """Generates a personalized greeting."""
    return f"Hello, {name}! Nice to meet you."

if __name__ == "__main__":
    mcp_server.run(transport="http", host="127.0.0.1", port=8000)
```

Скрипт определяет функцию `greet`, регистрируемую как MCP-инструмент. Декоратор `@tool()` автоматически регистрирует её. Docstring и аннотации типов формируют описание для LLM. Сервер слушает на `localhost:8000`.

## Краткий обзор

**Что:** LLM должны выходить за рамки генерации текста для взаимодействия с внешним миром. Без стандарта каждая интеграция — кастомная, сложная и непереиспользуемая.

**Почему:** MCP — универсальный интерфейс клиент-серверной архитектуры. Серверы экспонируют инструменты, ресурсы и промпты совместимым клиентам. Стандартизация создаёт экосистему интероперабельных компонентов.

**Когда использовать:** При построении сложных, масштабируемых агентных систем с разнообразными внешними инструментами и API.
