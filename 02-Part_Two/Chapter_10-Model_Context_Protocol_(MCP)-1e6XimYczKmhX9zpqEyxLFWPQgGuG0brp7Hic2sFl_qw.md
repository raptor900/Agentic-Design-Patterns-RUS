# Глава 10: Протокол контекста модели (MCP)

Чтобы LLM эффективно функционировали как агенты, их возможности должны выходить за рамки мультимодальной генерации. Необходимо взаимодействие с внешней средой, включая доступ к актуальным данным, использование внешнего программного обеспечения и выполнение конкретных операционных задач. Протокол контекста модели (Model Context Protocol, MCP) решает эту потребность, предоставляя стандартизированный интерфейс для взаимодействия LLM с внешними ресурсами. Этот протокол служит ключевым механизмом для обеспечения последовательной и предсказуемой интеграции.

## Обзор паттерна MCP

Представьте себе универсальный адаптер, позволяющий любой LLM подключиться к любой внешней системе, базе данных или инструменту без кастомной интеграции для каждого. По сути, именно это и представляет собой Протокол контекста модели (MCP). Это открытый стандарт, предназначенный для унификации того, как LLM — такие как Gemini, модели GPT от OpenAI, Mixtral и Claude — взаимодействуют с внешними приложениями, источниками данных и инструментами. Можно представить его как универсальный механизм подключения, упрощающий получение контекста, выполнение действий и взаимодействие LLM с различными системами.

MCP работает по клиент-серверной архитектуре. Он определяет, как различные элементы — данные (называемые resources), интерактивные шаблоны (по сути промпты) и исполняемые функции (известные как tools) — экспонируются MCP-сервером. Затем они потребляются MCP-клиентом, которым может быть LLM-хост-приложение или сам AI-агент. Этот стандартизированный подход значительно снижает сложность интеграции LLM в разнообразные операционные среды.

Однако MCP является контрактом для «агентного интерфейса», и его эффективность во многом зависит от дизайна underlying API, которые он экспонирует. Существует риск, что разработчики просто оборачивают существующие legacy API без модификации, что может быть субоптимально для агента. Например, если API системы тикетов позволяет извлекать только полные данные по одному тикету за раз, агенту, которому поручено обобщить тикеты высокого приоритета, будет медленно и неточно при больших объёмах. Для истинной эффективности underlying API должен быть улучшен детерминированными функциями, такими как фильтрация и сортировка, чтобы помочь недетерминированному агенту работать эффективно. Это подчёркивает, что агенты не заменяют детерминированные рабочие процессы магическим образом; они часто требуют более сильной детерминированной поддержки для успеха.

Более того, MCP может оборачивать API, вход или выход которого по-прежнему не является inherently понятным для агента. API полезен только тогда, когда его формат данных является дружественным к агенту — гарантия, которую сам MCP не обеспечивает. Например, создание MCP-сервера для хранилища документов, возвращающего файлы в формате PDF, практически бесполезно, если потребляющий агент не может распарсить содержимое PDF. Лучший подход — сначала создать API, возвращающий текстовую версию документа, например в формате Markdown, которую агент может действительно прочитать и обработать. Это демонстрирует, что разработчики должны учитывать не только подключение, но и природу обмениваемых данных для обеспечения истинной совместимости.
```python
import os

from google.adk.agents import LlmAgent
from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, StdioServerParameters


# Create a reliable absolute path to a folder named 'mcp_managed_files'
# within the same directory as this agent script.
# This ensures the agent works out-of-the-box for demonstration.
# For production, you would point this to a more persistent and secure location.
TARGET_FOLDER_PATH = os.path.join(
    os.path.dirname(os.path.abspath(__file__)),
    "mcp_managed_files",
)

# Ensure the target directory exists before the agent needs it.
os.makedirs(TARGET_FOLDER_PATH, exist_ok=True)

root_agent = LlmAgent(
    model="gemini-2.0-flash",
    name="filesystem_assistant_agent",
    instruction=(
        "Help the user manage their files. You can list files, read files, and write files. "
        f"You are operating in the following directory: {TARGET_FOLDER_PATH}"
    ),
    tools=[
        MCPToolset(
            connection_params=StdioServerParameters(
                command="npx",
                args=[
                    "-y",  # Argument for npx to auto-confirm install
                    "@modelcontextprotocol/server-filesystem",
                    # This MUST be an absolute path to a folder.
                    TARGET_FOLDER_PATH,
                ],
            ),
            # Optional: You can filter which tools from the MCP server are exposed.
            # For example, to only allow reading:
            # tool_filter=['list_directory', 'read_file']
        )
    ],
)
```

## MCP vs. Вызов функций инструментов

Протокол контекста модели (MCP) и вызов функций инструментов — различные механизмы, позволяющие LLM взаимодействовать с внешними возможностями (включая инструменты) и выполнять действия. Хотя оба служат для расширения возможностей LLM за пределы генерации текста, они различаются подходом и уровнем абстракции.

Вызов функций инструментов можно рассматривать как прямой запрос от LLM к конкретному, предопределённому инструменту или функции. В этом контексте мы используем слова «инструмент» и «функция» как взаимозаменяемые. Это взаимодействие характеризуется моделью коммуникации «один-к-одному», где LLM форматирует запрос на основе понимания намерения пользователя, требующего внешнего действия. Затем код приложения выполняет этот запрос и возвращает результат LLM. Этот процесс часто является проприетарным и различается у разных провайдеров LLM.

В противоположность этому, Протокол контекста модели (MCP) действует как стандартизированный интерфейс для обнаружения, коммуникации и использования внешних возможностей LLM. Он функционирует как открытый протокол, facilitating взаимодействие с широким спектром инструментов и систем, стремясь создать экосистему, в которой любой совместимый инструмент может быть доступен любой совместимой LLM. Это способствует интероперабельности, композируемости и переиспользуемости между различными системами и реализациями. Adopting a federated model, мы значительно улучшаем интероперабельность и раскрываем ценность существующих активов. Эта стратегия позволяет интегрировать разрозненные и legacy-сервисы в современную экосистему, simply оборачивая их в MCP-совместимый интерфейс. Эти сервисы продолжают работать независимо, но теперь могут композироваться в новые приложения и рабочие процессы, с их сотрудничеством, orchestrated LLM. Это способствует гибкости и переиспользуемости без необходимости costly переписывания фундаментальных систем.

Вот разбивка фундаментальных различий между MCP и вызовом функций инструментов:

| Возможность | Вызов функций инструментов | MCP |
|---|---|---|
| **Стандартизация** | Проприетарный, зависит от вендора. Формат и реализация различаются у провайдеров LLM. | Открытый стандартизированный протокол, promoting интероперабельность между различными LLM и инструментами. |
| **Область** | Прямой механизм для LLM запросить выполнение конкретной предопределённой функции. | Более широкий фреймворк для обнаружения и коммуникации LLM и внешних инструментов. |
| **Архитектура** | Взаимодействие «один-к-одному» между LLM и логикой обработки инструментов приложения. | Клиент-серверная архитектура, где LLM-приложения (клиенты) подключаются и используют различные MCP-серверы (инструменты). |
| **Обнаружение** | LLM явно told о доступных инструментах в контексте конкретного разговора. | Обеспечивает динамическое обнаружение доступных инструментов. MCP-клиент может запросить сервер о его возможностях. |
| **Переиспользуемость** | Интеграции инструментов tightly coupled с конкретным приложением и LLM. | Способствует разработке переиспользуемых автономных MCP-серверов, доступных любому совместимому приложению. |

Вызов функций — как набор кастомных инструментов для конкретной мастерской с фиксированным набором задач. MCP — как создание универсальной стандартизированной системы электрических розеток. Он не предоставляет инструменты, но позволяет любому совместимому инструменту подключаться и работать, enabling динамически расширяющуюся мастерскую.

Кратко: вызов функций даёт прямой доступ к нескольким конкретным функциям, MCP — стандартизированный коммуникационный фреймворк, позволяющий LLM обнаруживать и использовать обширный спектр внешних ресурсов. Для простых приложений достаточно конкретных инструментов; для сложных, взаимосвязанных AI-систем, нуждающихся в адаптации, необходим универсальный стандарт вроде MCP.

## Дополнительные соображения по MCP

Хотя MCP представляет собой мощный фреймворк, thorough оценка требует учёта нескольких аспектов, влияющих на его пригодность:

* **Tool vs. Resource vs. Prompt:** Важно понимать специфические роли этих компонентов. Resource — статические данные (например, PDF-файл, запись базы данных). Tool — исполняемая функция, выполняющая действие (отправка письма, запрос к API). Prompt — шаблон, направляющий LLM во взаимодействии с ресурсом или инструментом, обеспечивающий структурированность и эффективность.
* **Обнаруживаемость (Discoverability):** Ключевое преимущество MCP — MCP-клиент может динамически запрашивать сервер о доступных инструментах и ресурсах. Этот механизм «just-in-time» discovery мощен для агентов, которым необходимо адаптироваться к новым возможностям без переразвёртывания.
* **Безопасность:** Экспонирование инструментов и данных через любой протокол требует надёжных мер безопасности. Реализация MCP должна включать аутентификацию и авторизацию для контроля доступа клиентов к серверам и permitted действий.
* **Реализация:** Хотя MCP — открытый стандарт, его реализация может быть сложной. Однако поставщики упрощают этот процесс: Anthropic и FastMCP предлагают SDK, абстрагирующие boilerplate-код.
* **Обработка ошибок:** Критически важна comprehensive стратегия. Протокол должен определять, как ошибки (сбой инструмента, недоступный сервер, невалидный запрос) передаются LLM для понимания и попытки альтернативного подхода.
* **Локальный vs. удалённый сервер:** MCP-серверы могут развёртываться локально (скорость и безопасность sensitive данных) или удалённо (масштабируемый общий доступ к инструментам).
* **По запросу vs. пакетная обработка:** MCP поддерживает оба режима: интерактивные сессии в реальном времени и пакетную обработку данных.
* **Транспортный механизм:** Протокол определяет транспортные слои. Для локальных взаимодействий — JSON-RPC через STDIO. Для удалённых — Streamable HTTP и Server-Sent Events (SSE) для persistent и эффективной клиент-серверной коммуникации.

MCP использует клиент-серверную модель для стандартизации потока информации. Понимание взаимодействия компонентов ключево для продвинутого агентного поведения:

1. **Большая языковая модель (LLM):** Основной интеллект. Обрабатывает запросы пользователя, формулирует планы, решает, когда обратиться к внешней информации или выполнить действие.
2. **MCP-клиент:** Приложение или обёртка вокруг LLM. Выступает посредником, переводя намерение LLM в формальный запрос, соответствующий стандарту MCP. Ответственен за обнаружение, подключение и коммуникацию с MCP-серверами.
3. **MCP-сервер:** Шлюз к внешнему миру. Экспонирует набор инструментов, ресурсов и промптов авторизованному MCP-клиенту. Каждый сервер обычно отвечает за конкретный домен: подключение к внутренней БД, email-сервис или публичный API.
4. **Опциональный сторонний сервис:** Фактический внешний инструмент, приложение или источник данных, которым MCP-сервер управляет и экспонирует. Конечная точка, выполняющая запрошенное действие.

Поток взаимодействия:

1. **Обнаружение:** MCP-клиент от имени LLM запрашивает MCP-сервер о доступных возможностях. Сервер отвечает manifest со списком инструментов, ресурсов и промптов.
2. **Формулирование запроса:** LLM определяет необходимость использования одного из обнаруженных инструментов (например, отправить письмо). Формулирует запрос с указанием инструмента и параметров.
3. **Коммуникация клиента:** MCP-клиент отправляет стандартизированный вызов соответствующему MCP-серверу.
4. **Выполнение на сервере:** MCP-сервер аутентифицирует клиента, валидирует запрос и выполняет действие через underlying software.
5. **Ответ и обновление контекста:** Сервер отправляет стандартизированный ответ клиенту. Клиент передаёт результат LLM, обновляя контекст для следующего шага.
```python
# ./adk_agent_samples/mcp_agent/__init__.py 
from . import agent
```

## Практические применения и сценарии использования

MCP значительно расширяет возможности AI/LLM, делая их более универсальными и мощными. Вот девять ключевых сценариев:

* **Интеграция с базами данных:** MCP позволяет LLM и агентам бесшовно взаимодействовать со структурированными данными. Например, через MCP Toolbox for Databases агент может запрашивать Google BigQuery для получения данных в реальном времени, генерации отчётов или обновления записей через естественный язык.
* **Оркестрация генеративных медиа:** MCP позволяет интегрироваться с сервисами генеративных медиа. Через MCP Tools for Genmedia Services агент может оркестрировать workflow с Google Imagen (генерация изображений), Veo (видео), Chirp 3 HD (реалистичные голоса), Lyria (композиция музыки).
* **Взаимодействие с внешними API:** MCP предоставляет стандартизированный способ вызова и получения ответов от любого внешнего API. Агент может получать погоду, биржевые котировки, отправлять письма, взаимодействовать с CRM.
* **Извлечение информации на основе рассуждений:** Используя reasoning-способности LLM, MCP facilitates эффективное, зависящее от запроса извлечение информации, превосходящее traditional поисковые системы. Вместо возврата целого документа агент анализирует текст и извлекает точную клаузу, цифру или утверждение.
* **Разработка пользовательских инструментов:** Разработчики могут создавать кастомные инструменты и экспонировать их через MCP-сервер (например, через FastMCP). Специализированные внутренние функции становятся доступны LLM в стандартизированном формате.
* **Стандартизированная коммуникация LLM-приложение:** MCP обеспечивает consistent коммуникационный слой. Снижает накладные расходы на интеграцию, promoting интероперабельность.
* **Оркестрация сложных рабочих процессов:** Комбинируя различные MCP-инструменты и источники данных, агенты могут оркестровать сложные многошаговые workflow. Например: извлечь данные клиента из БД → сгенерировать персонализированное изображение → составить email → отправить.
* **Управление IoT-устройствами:** MCP может facilitating взаимодействие LLM с устройствами Интернета вещей. Агент может отправлять команды к умной бытовой технике, промышленным сенсорам, роботам.
* **Автоматизация финансовых сервисов:** В финтехе MCP позволяет LLM взаимодействовать с финансовыми источниками данных, торговыми платформами, compliance-системами. Агент может анализировать рыночные данные, совершать сделки, генерировать персонализированные советы, автоматизировать регуляторную отчётность.

Кратко: MCP позволяет агентам получать доступ к данным в реальном времени из БД, API и веб-ресурсов. Также позволяет выполнять действия: отправлять письма, обновлять записи, управлять устройствами, выполнять сложные задачи. MCP поддерживает инструменты генерации медиа для AI-приложений.

## Практический пример кода с ADK

Этот раздел описывает подключение к локальному MCP-серверу файловой системы, enabling агенту ADK взаимодействие с локальной файловой системой.

### Настройка агента с MCPToolset

Для настройки агента необходимо создать файл agent.py (например, в ./adk_agent_samples/mcp_agent/agent.py). MCPToolset инстанцируется в списке tools объекта LlmAgent. Критически важно заменить "/path/to/your/folder" в args абсолютным путём к директории на локальной системе, к которой MCP-сервер может получить доступ.

```python
connection_params = StdioConnectionParams(
    server_params={
        "command": "python3",
        "args": ["./agent/mcp_server.py"],
        "env": {
            "SERVICE_ACCOUNT_PATH": SERVICE_ACCOUNT_PATH,
            "DRIVE_FOLDER_ID": DRIVE_FOLDER_ID,
        },
    }
)
```

npx (Node Package Execute), входящий в npm (Node Package Manager) версий 5.2.0 и выше — утилита для прямого выполнения Node.js-пакетов из реестра npm. Это устраняет необходимость глобальной установки. По сути, npx — npm package runner, commonly используемый для запуска community MCP-серверов, distributed как Node.js-пакеты.

Создание файла __init__.py необходимо для recognition agent.py как части discoverable Python-пакета для ADK. Этот файл должен находиться в той же директории, что и agent.py.

```python
# ./adk_agent_samples/mcp_agent/__init__.py
from . import agent
```

Конечно, доступны и другие поддерживаемые команды. Например, подключение к python3:

```python
connection_params = StdioConnectionParams(
    server_params={
        "command": "uvx",
        "args": ["mcp-google-sheets@latest"],
        "env": {
            "SERVICE_ACCOUNT_PATH": SERVICE_ACCOUNT_PATH,
            "DRIVE_FOLDER_ID": DRIVE_FOLDER_ID,
        },
    }
)
```

UVX в контексте Python — командная строка, использующая uv для выполнения команд во временной изолированной Python-среде. Позволяет запускать Python-инструменты и пакеты без глобальной или project-level установки. Можно запускать через MCP-сервер.

```python
cd ./adk_agent_samples # Or your equivalent parent directory 
adk web
```После создания MCP-сервера следующий шаг — подключение к нему.

## Подключение MCP-сервера через ADK Web

Для начала выполните 'adk web'. Перейдите в родительскую директорию mcp_agent (например, adk_agent_samples) в терминале:

```python
cd ./adk_agent_samples # Или ваша эквивалентная родительская директория

## Создание MCP-сервера с FastMCP

FastMCP — высокоуровневый Python-фреймворк для упрощённой разработки MCP-серверов. Предоставляет абстрактный слой, упрощающий сложности протокола, позволяя разработчикам фокусироваться на core-логике.

Библиотека позволяет быстро определять инструменты, ресурсы и промпты с помощью простых Python-декораторов. Значительное преимущество — автоматическая генерация схем, intelligently интерпретирующая сигнатуры функций, аннотации типов и docstring для конструирования спецификаций интерфейсов AI-модели. Эта автоматизация минимизирует ручную конфигурацию и снижает human error.

Помимо базового создания инструментов, FastMCP facilitates продвинутые архитектурные паттерны: композиция серверов и проксирование. Это enables модульную разработку сложных многокомпонентных систем и бесшовную интеграцию существующих сервисов. FastMCP также включает оптимизации для эффективных, распределённых и масштабируемых AI-приложений.

```python
# ./adk_agent_samples/fastmcp_client_agent/agent.py
import os

from google.adk.agents import LlmAgent
from google.adk.tools.mcp_tool.mcp_toolset import MCPToolset, HttpServerParameters


# Define the FastMCP server's address.
# Make sure your fastmcp_server.py (defined previously) is running on this port.
FASTMCP_SERVER_URL = "http://localhost:8000"

root_agent = LlmAgent(
    model="gemini-2.0-flash",  # Or your preferred model
    name="fastmcp_greeter_agent",
    instruction='You are a friendly assistant that can greet people by their name. Use the "greet" tool.',
    tools=[
        MCPToolset(
            connection_params=HttpServerParameters(
                url=FASTMCP_SERVER_URL,
            ),
            # Optional: Filter which tools from the MCP server are exposed
            # For this example, we're expecting only 'greet'
            tool_filter=["greet"],
        )
    ],
)
```

Этот Python-скрипт определяет функцию greet, принимающую имя и возвращающую персонализированное приветствие. Декоратор @tool() автоматически регистрирует её как инструмент, который AI или другая программа может использовать. Docstring и аннотации типов используются FastMCP для описания инструмента агенту.

При запуске скрипт стартует FastMCP-сервер, listening на localhost:8000. Это делает функцию greet доступной как сетевой сервис. Агент затем может быть настроен для подключения к серверу и использования greet-инструмента. Сервер работает непрерывно до ручной остановки.

## Использование FastMCP-сервера с ADK-агентом

ADK-агент может быть настроен как MCP-клиент для использования FastMCP-сервера. Требуется конфигурация HttpServerParameters с адресом сервера (обычно http://localhost:8000).

Параметр tool_filter позволяет ограничить использование инструментов конкретными (например, 'greet'). Когда пользователь просит "Greet John Doe", встроенная LLM агента идентифицирует greet-инструмент через MCP, вызывает его с аргументом "John Doe" и возвращает ответ сервера.

Для настройки необходим файл agent.py (например, в ./adk_agent_samples/fastmcp_client_agent/), instancирующий ADK-агента с HttpServerParameters.
