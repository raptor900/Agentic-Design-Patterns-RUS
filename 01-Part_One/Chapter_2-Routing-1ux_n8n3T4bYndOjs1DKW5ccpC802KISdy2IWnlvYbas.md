# Глава 2: Маршрутизация

## Обзор паттерна «Маршрутизация»

Хотя последовательная обработка через цепочку промптов — базовая техника выполнения детерминированных линейных рабочих процессов с языковыми моделями, её применимость ограничена в сценариях, требующих адаптивных ответов. Реальные агентные системы часто должны выбирать между несколькими возможными действиями на основе контингентных факторов: состояния среды, пользовательского ввода или результата предыдущей операции. Эта способность к динамическому принятию решений, управляющая потоком управления между различными специализированными функциями, инструментами или подпроцессами, реализуется через механизм, известный как маршрутизация (routing).

Маршрутизация вносит условную логику в операционную систему агента, обеспечивая переход от фиксированного пути выполнения к модели, где агент динамически оценивает конкретные критерии для выбора из набора возможных последующих действий. Это позволяет системе работать более гибко и контекстно-ориентировано.

Например, агент для обработки клиентских обращений, оснащённый функцией маршрутизации, сначала классифицирует входящий запрос для определения намерения пользователя. На основе классификации он направляет запрос к специализированному агенту для прямого ответа, к инструменту извлечения данных из базы для информации об аккаунте или к процедуре эскалации для сложных вопросов. Более продвинутый агент с маршрутизацией может:

1. Проанализировать запрос пользователя.
2. **Направить** запрос на основе *намерения*:
   * Если намерение — «проверить статус заказа», направить к подагенту или цепочке инструментов для работы с базой заказов.
   * Если намерение — «информация о продукте», направить к подагенту или цепочке поиска по каталогу.
   * Если намерение — «техническая поддержка», направить к другой цепочке, обращающейся к руководствам по устранению неполадок или к человеку.
   * Если намерение неясно, направить к подагенту уточнения или цепочке промптов.

Базовый компонент паттерна маршрутизации — механизм, выполняющий оценку и направляющий поток. Этот механизм можно реализовать несколькими способами:

* **Маршрутизация на основе LLM:** Саму языковую модель можно prompting для анализа входа и вывода конкретного идентификатора или инструкции, указывающих следующий шаг или направление.
* **Маршрутизация на основе эмбеддингов:** Входной запрос преобразуется в векторное представление, которое сравнивается с эмбеддингами маршрутов. Запрос направляется к наиболее похожему маршруту — полезно для семантической маршрутизации.
* **Правиловая маршрутизация:** Используются предопределённые правила (if-else, switch) на основе ключевых слов, паттернов или структурированных данных. Быстрее и детерминированнее, но менее гибко.
* **Маршрутизация на основе ML-модели:** Используется дискриминативная модель (классификатор), специально обученная на размеченных данных. В отличие от LLM-маршрутизации, логика закодирована в обученных весах модели.

Механизмы маршрутизации могут применяться на разных этапах операционного цикла агента: в начале для классификации основной задачи, на промежуточных этапах цепочки для определения последующего действия или при выборе наиболее подходящего инструмента.

Фреймворки вроде LangChain, LangGraph и Google ADK предоставляют явные конструкции для определения и управления такой условной логикой. LangGraph с архитектурой на основе состояний особенно подходит для сложных сценариев маршрутизации. Реализация маршрутизации позволяет системе выйти за рамки детерминированной последовательной обработки к более адаптивным потокам выполнения.

## Практические применения и сценарии использования

Паттерн маршрутизации — критический механизм управления в дизайне адаптивных агентных систем.

В **человеко-компьютерном взаимодействии** (виртуальные ассистенты, AI-репетиторы) маршрутизация интерпретирует намерение пользователя и определяет наиболее подходящее действие.

В **конвейерах обработки данных** маршрутизация выступает функцией классификации и распределения: входящие данные (письма, тикеты, API-запросы) анализируются и направляются в соответствующий рабочий процесс.

В **сложных системах со специализированными инструментами** маршрутизация действует как диспетчер высокого уровня, назначающий задачи наиболее подходящему агенту.

## Практический пример кода (LangChain)

Следующий код демонстрирует простую агентную систему с использованием LangChain и Google Generative AI. «Координатор» маршрутизирует запросы пользователя к обработчикам на основе намерения.

```python
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough, RunnableBranch

# --- Конфигурация ---
try:
    llm = ChatGoogleGenerativeAI(model="gemini-2.5-flash", temperature=0)
    print(f"Языковая модель инициализирована: {llm.model}")
except Exception as e:
    print(f"Ошибка инициализации: {e}")
    llm = None

# --- Обработчики подагентов ---
def booking_handler(request: str) -> str:
    """Обработчик бронирования."""
    print("\n--- ПЕРЕДАЧА ОБРАБОТЧИКУ БРОНИРОВАНИЯ ---")
    return f"Обработчик бронирования обработал запрос: '{request}'. Результат: симуляция бронирования."

def info_handler(request: str) -> str:
    """Обработчик информационных запросов."""
    print("\n--- ПЕРЕДАЧА ИНФОРМАЦИОННОМУ ОБРАБОТЧИКУ ---")
    return f"Информационный обработчик обработал: '{request}'. Результат: симуляция извлечения информации."

def unclear_handler(request: str) -> str:
    """Обработчик неопределённых запросов."""
    print("\n--- ОБРАБОТКА НЕОПРЕДЕЛЁННОГО ЗАПРОСА ---")
    return f"Координатор не смог делегировать запрос: '{request}'. Уточните, пожалуйста."

# --- Цепочка маршрутизации координатора ---
coordinator_router_prompt = ChatPromptTemplate.from_messages([
    ("system",
     "Проанализируйте запрос пользователя и определите, какой обработчик должен его обработать.\n"
     "- Если запрос связан с бронированием, выводите 'booker'.\n"
     "- Для всех остальных информационных вопросов выводите 'info'.\n"
     "- Если запрос неясен, выводите 'unclear'.\n"
     "Выводите ТОЛЬКО одно слово: 'booker', 'info' или 'unclear'."),
    ("user", "{request}")
])

if llm:
    coordinator_router_chain = coordinator_router_prompt | llm | StrOutputParser()

# --- Логика делегирования ---
branches = {
    "booker": RunnablePassthrough.assign(
        output=lambda x: booking_handler(x['request']['request'])
    ),
    "info": RunnablePassthrough.assign(
        output=lambda x: info_handler(x['request']['request'])
    ),
    "unclear": RunnablePassthrough.assign(
        output=lambda x: unclear_handler(x['request']['request'])
    ),
}

delegation_branch = RunnableBranch(
    (lambda x: x['decision'].strip() == 'booker', branches["booker"]),
    (lambda x: x['decision'].strip() == 'info', branches["info"]),
    branches["unclear"]
)

coordinator_agent = {
    "decision": coordinator_router_chain,
    "request": RunnablePassthrough()
} | delegation_branch | (lambda x: x['output'])

def main():
    if not llm:
        print("\nПропуск выполнения из-за ошибки инициализации LLM.")
        return

    print("--- Запрос на бронирование ---")
    result_a = coordinator_agent.invoke({"request": "Забронируй мне рейс в Лондон."})
    print(f"Результат A: {result_a}")

    print("\n--- Информационный запрос ---")
    result_b = coordinator_agent.invoke({"request": "Какова столица Италии?"})
    print(f"Результат B: {result_b}")

    print("\n--- Неопределённый запрос ---")
    result_c = coordinator_agent.invoke({"request": "Расскажи о квантовой физике."})
    print(f"Результат C: {result_c}")

if __name__ == "__main__":
    main()
```

Этот код строит простую агентную систему на LangChain и Google Generative AI. Три обработчика: бронирования, информационный и неопределённый. Координатор с помощью LLM классифицирует запросы и делегирует соответствующему обработчику через RunnableBranch.

## Краткий обзор

**Что:** Агентные системы должны реагировать на разнообразные входные данные, не обрабатываемые одним линейным процессом.

**Почему:** Маршрутизация вносит условную логику, позволяя системе анализировать входящий запрос и динамически направлять поток управления к наиболее подходящему инструменту или подагенту.

**Когда использовать:** Когда агент должен выбирать между несколькими рабочими процессами, инструментами или подагентами на основе пользовательского ввода или текущего состояния.

## Ключевые выводы

* Маршрутизация позволяет агентам принимать динамические решения о следующем шаге на основе условий.
* Она позволяет обрабатывать разнообразные входные данные и адаптировать поведение.
* Логика маршрутизации может реализовываться через LLM, правиловые системы или семантическое сходство эмбеддингов.
* Фреймворки вроде LangGraph и Google ADK предоставляют структурированные способы управления маршрутизацией.

## Заключение

Паттерн маршрутизации — ключевой шаг к построению по-настоящему динамичных и отзывчивых агентных систем. Освоение этого паттерна необходимо для создания универсальных и надёжных агентных приложений, способных интеллектуально ориентироваться в различных сценариях.
