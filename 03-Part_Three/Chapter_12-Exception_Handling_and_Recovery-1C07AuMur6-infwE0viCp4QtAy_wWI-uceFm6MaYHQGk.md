# Глава 12: Обработка исключений и восстановление

Чтобы AI-агенты работали надёжно в различных реальных средах, они должны управлять непредвиденными ситуациями, ошибками и сбоями. Подобно тому как люди адаптируются к неожиданным препятствиям, интеллектуальным агентам нужны надёжные системы для обнаружения проблем, запуска процедур восстановления или как минимум обеспечения контролируемого отказа. Это требование составляет основу паттерна «Обработка исключений и восстановление».

Этот паттерн сосредоточен на разработке исключительно устойчивых и живучих агентов, способных поддерживать бесперебойную функциональность и операционную целостность, несмотря на различные трудности и аномалии. Он подчёркивает важность как проактивной подготовки, так и реактивных стратегий для обеспечения непрерывной работы.

Этот паттерн иногда используется совместно с рефлексией: если первая попытка неудачна и вызывает исключение, рефлекторный процесс анализирует провал и повторяет задачу с доработанным подходом (например, улучшенным промптом).

## Обзор паттерна «Обработка исключений и восстановление»

Паттерн решает задачу управления операционными сбоями. Он включает прогнозирование потенциальных проблем (ошибки инструментов, недоступность сервисов) и разработку стратегий их смягчения: логирование ошибок, повторные попытки, откаты, graceful degradation и уведомления. Кроме того, паттерн подчёркивает механизмы восстановления: откат состояния, диагностику, самокоррекцию и эскалацию.

![Ключевые компоненты обработки исключений для AI-агентов](../assets/Key_Components_of_Exception_Handling_and_Recovery_for_AI_agents.png)

Рис. 1: Ключевые компоненты обработки исключений и восстановления для AI-агентов

**Обнаружение ошибок:** Тщательная идентификация операционных проблем. Это может проявляться как некорректный вывод инструментов, API-ошибки (404, 500), аномально долгое время ответа или бессмысленные ответы. Дополнительно может использоваться мониторинг со стороны других агентов или специализированных систем.

**Обработка ошибок:** После обнаружения необходим тщательно продуманный план реагирования: запись деталей в логи для отладки, повторные попытки (особенно для транзиентных ошибок), альтернативные стратегии (fallback), поддержание частичной функциональности (graceful degradation), оповещение операторов или других агентов (notification).

**Восстановление:** Восстановление агента или системы в стабильное операционное состояние. Может включать откат недавних изменений, расследование причин, корректировку плана/логики/параметров (самокоррекция) или делегирование проблемы оператору (эскалация).

Реализация этого паттерна превращает хрупкие системы в надёжные компоненты, способные работать в сложных и непредсказуемых средах.

## Практические применения и сценарии использования

Обработка исключений критически важна для любого агента в реальных условиях.

* **Чатботы клиентской поддержки:** При недоступности базы данных бот не должен падать — он обнаруживает ошибку API, информирует пользователя, предлагает попозже или эскалирует оператору.
* **Автоматизированный трейдинг:** При ошибке «недостаточно средств» бот логирует, не повторяет невалидную сделку, уведомляет пользователя.
* **Умный дом:** При сбое управления светом из-за сетевой проблемы агент повторяет попытку, а при неудаче — уведомляет пользователя.
* **Агенты обработки данных:** При повреждённом файле — пропуск, логирование, продолжение обработки, отчёт о пропущенных файлах.
* **Веб-скрапинг:** При CAPTCHA, изменении структуры сайта или серверной ошибке — пауза, прокси, отчёт о проблемном URL.
* **Робототехника:** При неудачном захвате компонента — обнаружение через сенсоры, корректировка, повторная попытка, оповещение оператора.

## Практический пример кода (ADK)

Обработка исключений и восстановление жизненно важны для надёжности. Рассмотрим, например, ответ агента на неудачный вызов инструмента. Причины могут быть в некорректном вводе или проблемах внешнего сервиса.

```python
from google.adk.agents import Agent, SequentialAgent

# Агент 1: Пытается использовать основной инструмент.
primary_handler = Agent(
    name="primary_handler",
    model="gemini-2.0-flash-exp",
    instruction="""
    Your job is to get precise location information. Use the get_precise_location_info
    tool with the user's provided address.
    """,
    tools=[get_precise_location_info],
)

# Агент 2: Действует как fallback-обработчик, проверяя состояние.
fallback_handler = Agent(
    name="fallback_handler",
    model="gemini-2.0-flash-exp",
    instruction="""
    Check if the primary location lookup failed by looking at state["primary_location_failed"].
    - If it is True, extract the city from the user's original query and use the get_general_area_info tool.
    - If it is False, do nothing.
    """,
    tools=[get_general_area_info],
)

# Агент 3: Представляет финальный результат из состояния.
response_agent = Agent(
    name="response_agent",
    model="gemini-2.0-flash-exp",
    instruction="""
    Review the location information stored in state["location_result"]. Present this information
    clearly and concisely to the user. If state["location_result"] does not exist or is empty,
    apologize that you could not retrieve the location.
    """,
    tools=[],
)

# SequentialAgent гарантирует порядок выполнения обработчиков.
robust_location_agent = SequentialAgent(
    name="robust_location_agent",
    sub_agents=[primary_handler, fallback_handler, response_agent],
)
```

Код определяет надёжную систему определения местоположения с использованием SequentialAgent. `primary_handler` — первый агент, пытающийся получить точную информацию. `fallback_handler` — резервный, проверяющий состояние `primary_location_failed`. `response_agent` — финальный, представляющий результат пользователю. SequentialAgent гарантирует порядок выполнения.

## Краткий обзор

**Что:** AI-агенты в реальных средах不可避免地 сталкиваются с непредвиденными ситуациями. Без структурированного управления проблемами агенты хрупки и ненадёжны.

**Почему:** Паттерн «Обработка исключений и восстановление» обеспечивает устойчивость: обнаружение ошибок (мониторинг выводов инструментов, API-ответов), стратегии обработки (логирование, повторные попытки, fallback) и протоколы восстановления (откат, самокоррекция, эскалация).

**Когда использовать:** Для любого AI-агента в динамичной реальной среде, где сбои инструментов, сетевые проблемы и непредсказуемые входные данные возможны.
